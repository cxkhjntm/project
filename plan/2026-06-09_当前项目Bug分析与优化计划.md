## 2. 严重程度定义

- P0：会导致核心功能不可用、数据/密钥风险、桌面包无法运行，必须优先修复。
- P1：会导致主要流程不稳定或体验明显错误，应在 MVP 可用前修复。
- P2：不一定阻断演示，但会影响质量、维护性、扩展性和长期稳定性。
- P3：产品体验、文档一致性和后续增强项。

## 3. 已确认 Bug 与风险清单

### P0-1 API Key 默认未加密落库

位置：
- `backend/app/config.py:27`
- `backend/app/services/crypto.py:24`
- `backend/.env.example:12`
- `README.md:296`

现象：
- `Settings.encrypt_api_keys` 默认值为 `False`。
- `.env.example` 只提供 `ENCRYPTION_KEY=`，没有启用加密的开关。
- `CryptoService` 在关闭加密时直接返回明文，导致 `Provider.api_key_encrypted` 字段实际可能存储明文。
- README 声称 API Key 会加密存储，并且把 Fernet 描述为 AES-256-GCM，这与实现不一致。

影响：
- 违背 AGENTS.md 中“API Key 加密存储、不得明文落库”的核心架构决策。
- 本地 SQLite 泄露时 API Key 直接暴露。

修复方向：
- 将默认值改为加密开启，新增显式配置 `ENCRYPT_API_KEYS=true`。
- 保留兼容读取：若解密失败且历史数据疑似明文，只在受控迁移路径中转换，不在普通读取中静默当作明文。
- README 改为准确表述 Fernet 机制，不再写 AES-256-GCM。
- 增加测试：创建 Provider 后数据库字段不等于原始 key，读取响应只返回 masked key。

### P0-2 讨论启动状态机允许重复启动

位置：
- `backend/app/routers/discussion.py:28`
- `backend/app/routers/discussion.py:58`
- `backend/app/routers/discussion.py:254`
- `frontend/src/hooks/useDiscussionSSE.ts`

现象：
- `GET /api/rooms/{room_id}/start` 允许 `running` 状态再次 start。
- `/control` 的 `start` 又不允许 `running`，两个入口规则不一致。
- 后端测试已有失败：`test_start_discussion_invalid_status` 期望 running 不可再次启动，但实际未抛错。
- 前端 EventSource 重连时会重新连接 `/start`，在 running 可再次启动的情况下可能创建重复后台讨论任务。

影响：
- 同一个房间可能并发跑多轮讨论、重复扣费、重复写消息、重复生成产物。
- 暂停/恢复/重连行为不可预测。

修复方向：
- 定义唯一状态机：`draft -> running -> paused -> completed | failed | stopped`。
- `running` 状态禁止再次启动；若只是恢复 SSE，应提供只读事件流端点，例如 `GET /events`。
- 把“启动讨论”和“订阅事件流”拆开：`POST /control start` 负责创建任务，`GET /events` 负责订阅。
- 增加房间级任务锁或 job registry，保证同一 room 同时只有一个 active task。

### P0-3 暂停/停止只改状态，不会控制正在运行的 Orchestrator

位置：
- `backend/app/routers/discussion.py:257-268`
- `backend/app/services/orchestrator.py`

现象：
- `/control pause|stop` 只修改数据库 `room.status`。
- `Orchestrator.run_discussion()` 不读取房间最新状态，也没有取消当前 LLM streaming 请求。
- stop 把状态写成 `completed`，后台任务仍可能继续发言、生成 artifact，并最终覆盖状态。

影响：
- 用户点击暂停或停止后，模型请求仍继续消耗 token。
- UI 展示的状态与真实后台任务不一致。

修复方向：
- 为每个 running room 持有 `asyncio.Task` 和 cancellation token。
- stop 调用应取消任务并关闭 EventSource，最终状态建议为 `stopped` 或 `cancelled`，不要伪装成 `completed`。
- pause 至少在每个专家回合之间检查状态；若要中断当前 streaming，需要将 httpx stream 包在可取消任务中。
- 增加集成测试：启动后 stop，不再新增消息和 artifact。

### P0-4 用户插话接口缺失

位置：
- `frontend/src/api/client.ts:161`
- `frontend/src/pages/DiscussionPage.tsx:173`
- `backend/app/routers/discussion.py:121`

现象：
- 前端 `sendRoomMessage()` 调用 `POST /api/rooms/{room_id}/messages`。

- Orchestrator 在每回合开始前读取最新用户消息，并将其加入 prompt。
- SSE 对用户消息也发 `message` 事件，前端不用额外刷新。

### P0-5 文件夹共享资料不会进入讨论上下文

位置：
- `backend/app/routers/sources.py:49`
- `backend/app/services/file_ingestion.py:109`
- `backend/app/services/orchestrator.py:115`
- `backend/app/services/context_builder.py:380`

现象：
- folder source 创建时只保存 `path` 和 `file_count`，不保存内容。
- Orchestrator 的 `load_shared_sources()` 只把数据库中的 `content` 传给 `ContextBuilder`。
- `ContextBuilder` 遇到无 `content` 的 source 会跳过。
- 因此“指定文件夹作为共享资料”即使添加成功，也不会真正注入专家 prompt。
- 另一个问题是 `sources.py` 使用 `validate_path(path, os.getcwd())`，会拒绝项目目录外的真实用户文件夹，与本地优先工具的设计矛盾。

影响：
- 核心场景“选择本地项目文件夹让专家分析”不可用或只显示文件数量。

修复方向：
- 移除“必须位于后端 cwd 内”的限制，改为绝对路径安全校验、排除目录、扩展名白名单、大小限制和用户确认。
- folder source 应建立轻量索引：文件清单、摘要、按需片段检索，而不是只存 file_count。
- Orchestrator 每轮按预算读取相关片段；MVP 可先使用 `scan_directory + read_file_content` 聚合前 N 个文件。
- 增加测试：添加临时目录 source 后，构建 prompt 中能看到目录文件内容。

### P0-6 Tauri 桌面包当前不可可靠运行

位置：
- `src-tauri/src/main.rs:50`
- `src-tauri/tauri.conf.json:58-68`
- `frontend/src/api/client.ts:6`

现象：
- Rust 健康检查请求 `/health`，FastAPI 实际端点是 `/api/health`。
- `tauri.conf.json` 引用 `icons/*` 和 `backend/expert-room-backend`，但 `src-tauri` 下缺少这些资源。
- 前端 API 基址固定为 `/api`，开发时依赖 Vite proxy；Tauri 生产包没有 Vite proxy，除非另行注入 backend base URL，否则会请求不到后端。

影响：
- 桌面壳可能无法打包，或打包后前端连不上 sidecar 后端。

修复方向：
- 补齐图标和 sidecar 构建流程。
- `check_backend_health()` 改为 `/api/health`。
- 前端 API base 支持环境变量和 Tauri 运行时配置：开发为 `/api`，桌面生产为 `http://127.0.0.1:<port>/api`。
- sidecar 启动后应等待健康检查通过再开放 UI 主流程。

### P1-1 手动生成产物可能 422，自动产物质量偏模板化

位置：
- `frontend/src/api/artifacts.ts:12`
- `backend/app/routers/artifacts.py:34`
- `backend/app/schemas/artifact.py:45`
- `backend/app/services/artifact_writer.py:118`

现象：
- 前端手动 `synthesize(roomId)` 不带 body。
- 后端 `request: SynthesizeRequest` 没有默认值，空 body 容易被 FastAPI 判为 422。
- `SynthesizeRequest.artifact_type/include_citations/max_length` 基本未被使用。
- `ArtifactWriter` 主要是模板拼接，`discussion[:2000]` 截断后填入固定章节，不是真正的综合生成。

影响：
- “讨论完成后生成高质量结构化文档”的核心价值不足。
- 手动 fallback 可能不可用。

修复方向：
- 前端至少发送 `{}`，或后端参数写为 `request: SynthesizeRequest = SynthesizeRequest()`。
- 将 `artifact_type/include_citations/max_length` 接入生成逻辑。
- 用 `ContextBuilder.build_synthesizer_prompt()` 调 LLM 生成最终产物，失败时再 fallback 到模板摘要。

### P1-2 前端错误处理丢失后端真实错误信息

位置：
- `backend/app/middleware/error_handler.py`
- `frontend/src/api/client.ts:59`
- `frontend/src/api/client.ts:188`

现象：
- 后端错误响应格式是 `{ error, message, request_id }`。
- 前端只读取 `error.detail`，多数情况下展示 `An error occurred` 或 `Upload failed`。

影响：
- 用户无法知道失败原因，例如路径越界、房间不存在、文件过大、schema 校验失败。

修复方向：
- ApiClient 错误解析兼容 `detail`、`message`、FastAPI 422 数组。
- 把 `request_id` 保留到 ApiError，方便排查。

### P1-4 主持人 ACTION 指令没有真正控制流程

位置：
- `backend/app/services/orchestrator.py`
- `backend/app/services/context_builder.py`

现象：
- 主持人 prompt 要求输出 `ACTION: next:<专家名称>`。
- Orchestrator 只处理 `converge` 和 `synthesize`，未按 `next` 选择发言专家。
- 实际每轮仍然遍历所有 participants。

影响：
- 主持人调度形同虚设。
- 快速模式、严格评审模式、定向追问都难落地。

修复方向：
- 若 action 为 `next`，下一步只运行指定专家。
- 标准模式可以由策略决定：全员轮询、主持人点名、用户指定专家。
- 增加测试：host 输出 `ACTION: next:后端工程师` 时只触发该专家发言。

### P1-5 滚动摘要在第二轮后可能停止更新

位置：
- `backend/app/services/message_service.py:59`
- `backend/app/services/orchestrator.py:455`
- `backend/app/services/context_builder.py`

现象：
- `message_service.get_by_room()` 默认按 round 升序。
- `_update_rolling_summary()` 取 `limit=len(participants)+1` 后再筛选 `current_round`。
- 第二轮开始时 limit 很可能取到第一轮消息，筛选当前轮后为空。

影响：
- 后续轮次 prompt 可能看不到最新讨论，导致重复发散。

修复方向：
- 按 `current_round` 直接查询消息，或按时间倒序取最近消息后再排序。
- 滚动摘要应保留关键决策、分歧、待办，而不是简单拼接 200 字片段。

### P2-2 后端 Ruff 与 Pydantic V2 迁移未完成

## 4. 建议修复路线

### 阶段 0：建立保护线

目标：先让问题可重复验证，避免边修边引入回归。

任务：
1. 新建或更新测试覆盖：
   - running 房间禁止重复 start。
   - `/control stop` 后不再新增消息和 artifact。
   - `POST /messages` 用户消息可写入并被 SSE/历史消息读取。
   - folder source 的文件内容进入 prompt。
   - Provider key 加密落库。
   - ApiClient 能显示后端 `message`。
2. 记录当前 dirty worktree，避免覆盖已有用户变更。
3. 将 `project/plan/00-03` 标注状态，避免继续执行过期项。

验收：
- 新增失败测试能准确复现当前 bug。

### 阶段 1：修复 P0 核心阻断

目标：核心闭环可用且不会重复扣费或泄露密钥。

任务：
1. Provider API Key 默认加密，修正 README 和 `.env.example`。
2. 重构讨论启动和 SSE 订阅：
   - `POST /control start` 只负责启动。
   - `GET /events` 只负责订阅。
   - 禁止 running 再启动。
3. 实现 stop/cancel/pause 的真实任务控制。
4. 新增用户消息 POST API，并让 Orchestrator 读取最新用户消息。
5. 修复 folder source 的扫描、索引和 prompt 注入。
6. 修复 Tauri health endpoint、API base、缺失资源和 sidecar 构建路径。

验收：
- `pytest` 全绿。
- 本地跑一次“创建 Provider -> 创建房间 -> 添加文件夹 -> 开始讨论 -> 用户插话 -> 停止/完成 -> 生成产物”闭环。
- Tauri 至少 `cargo check` 通过；正式包构建路径明确。

### 阶段 2：修复 P1 体验与契约问题

目标：用户看到的错误、状态、产物和成本数据可信。

任务：
1. 前端 ApiClient 兼容 `{message}` 和 422 错误数组。
2. 手动 synthesize 发送 `{}` 或后端设置默认 request。
3. ArtifactWriter 接入 LLM synthesizer prompt，并保留 deterministic fallback。
4. 统一前后端 round limit。
5. 主持人 `ACTION: next` 真正控制发言。
6. 修复 rolling summary 查询当前轮的 bug。
7. 实现 token/cost update。
8. 同步 TypeScript 状态枚举和 source type。

验收：
- 前端 build/lint 通过。
- 用户错误提示能显示具体后端 message。
- 讨论中 token 统计非 0 或明确显示“估算”。
- 产物内容包含真实讨论结论，不只是固定模板。

### 阶段 3：工程质量与维护性
