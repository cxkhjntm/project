# 02. 上下文与编排逻辑优化方案 (详细)

## 1. 废弃暴力的正则收敛 (`backend/app/services/orchestrator.py`)
原有的 `check_convergence` 通过 `CONVERGENCE_KEYWORDS` 进行暴力匹配。遇到如“这个方案基本可行，但在并发上会有瓶颈”的表述时，会被误判为收敛，强行终止讨论。

**改造步骤：**
完全删除 `CONVERGENCE_KEYWORDS` 与 `check_convergence` 函数。
在 `_run_orchestrator_turn` 方法中，依赖主持人 (Orchestrator) 自身的智能判定：
```python
# 提取 action
action = parse_host_action(host_content)

# 当主持人决定所有观点一致且不再需要补充时，它会输出 ACTION: converge
if action and action.get("type") == "converge":
    # 直接跳出讨论循环，进入生成产物阶段
    break
```

## 2. 强化主持人的 Prompt 指令 (`backend/app/services/context_builder.py`)
我们需要在 `build_orchestrator_prompt` 中明确教导主持人如何下达闭环指令。

**修改 Prompt：**
```python
prompt = f"""你是专家群聊主持人。你的任务是控制讨论流程。

...
你需要：
1. 根据任务目标安排专家发言顺序（在结尾输出 ACTION: next:<专家名字>）
2. 识别讨论中的冲突、遗漏和风险
3. 【关键】当评估所有专家的论点已经补齐并且不存在矛盾，或者方案已经可以落地时，立即停止讨论，输出指令：`ACTION: converge`。
4. 在输出 converge 前，可以做一两句简短的最终总结。

请用简洁的主持词引导，不要超过 200 字。
"""
```

## 3. 改善“滚动摘要”的粗暴截断机制 (`backend/app/services/context_builder.py`)
当前的 `build_rolling_summary` 在字数超出时，直接使用 `combined[-max_chars:]` 从后往前硬切，这会导致句子断裂，且旧共识被完全抛弃。

**改进方案（滑动窗口队列）：**
在 `orchestrator.py` 中不要仅仅传字符串。将历史记录按“消息实体”传入，并在 `ContextBuilder` 中处理优先级。
```python
def build_messages_context(self, all_messages: List[Dict[str, Any]], max_tokens: int = 4000) -> str:
    """更智能的上下文组装器，优先保留第一轮的目标设定和最近的 N 轮"""
    if not all_messages:
        return "讨论刚开始。"
        
    parts = []
    # 强制保留第 1 轮的内容（通常包含基调设定）
    first_round = [m for m in all_messages if m.get("round") == 1]
    
    # 收集最近的有效信息，从后往前倒推
    recent_messages = []
    current_tokens = 0
    token_limit_for_recent = int(max_tokens * self.chars_per_token)
    
    for msg in reversed(all_messages):
        # 排除已经是 first_round 的消息
        if msg.get("round") == 1:
            continue
            
        content = f"[{msg.get('sender_id')}]: {msg.get('content')}"
        if current_tokens + len(content) > token_limit_for_recent:
            break
            
        recent_messages.insert(0, content)
        current_tokens += len(content)
        
    # 拼接
    parts.append("【初始观点设定】")
    for msg in first_round:
         parts.append(f"[{msg.get('sender_id')}]: {msg.get('content')}")
         
    parts.append("\n【近期讨论纪要】")
    parts.extend(recent_messages)
    
    return "\n".join(parts)
```
然后在 `_update_rolling_summary` 中不再做粗暴的 string replace，而是每轮动态构建 `build_messages_context` 送入 Prompt。这避免了文本摘要越来越长且断行错乱的问题。
