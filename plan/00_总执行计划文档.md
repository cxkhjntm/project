# 专家团(Expert Room)源码分析与总执行计划

## 一、 分析背景
本项目是一个本地优先的多 AI 专家协作讨论工作台，采用 FastAPI 后端和 React (Vite) 前端。核心功能围绕：房间管理、角色卡定义、讨论编排以及产物生成。
经过代码走查，确认整体框架完整，功能逻辑清晰。然而，考虑到本项目为**个人本地使用**，在体验细节（特别是流式输出）、架构复杂度（数据库与安全）以及上下文管理方面，存在不小优化空间。

## 二、 核心缺陷与问题汇总

### 1. 核心体验问题 (严重缺陷)
- **缺乏真正的流式响应 (SSE Streaming)**：后端 `orchestrator.py` 在等待 LLM 返回时是完全阻塞的，直到拿到了完整的字符串才通过 SSE 发送一条 `MESSAGE`。这使得前端在 10-30 秒内只能展示“思考中”，用户体验大打折扣，缺乏真实的 AI 讨论感。
- **并发与网络开销**：`model_client.py` 中，每一次 LLM 请求都重新实例化 `httpx.AsyncClient`，缺乏连接复用，对于本地多专家并发讨论场景会带来无谓的延迟。
- **长讨论的失控风险**：缺乏真正的“打断(Stop)”或“取消(Cancel)”能力，若某专家陷入长篇大论，无法立即停止其 API 调用。

### 2. 针对“个人使用”的过度设计
- **数据库迁移沉重**：使用了 SQLAlchemy + Alembic。对于个人项目而言，每次添加一个小字段都要跑迁移脚本过于繁重，完全可以依赖 SQLAlchemy 在启动时的 `create_all()` 配合 SQLite 完成。
- **过度加密保护**：API Key 使用 Fernet 强制加密，虽然安全，但在本地个人工具中这极大地提高了备份和迁移门槛。需要增加明文 `.env` 备用读取机制或可视化导出功能。

### 3. 上下文记忆机制缺陷
- **“滚动摘要”实现简单粗暴**：`_update_rolling_summary` 目前仅合并上一轮的消息，若轮数较多，容易丢失早期的关键决策和约束。需要基于 Token 窗口进行智能裁剪，而不是简单按轮次截断。
- **收敛算法不够健壮**：仅通过关键词（如“可行”、“同意”）进行正则匹配来判断讨论是否收敛，容易产生误判。
